# Engineering Knowledge Base

## Working Principles

> These principles define how this Engineering Knowledge Base is created, maintained, and evolved.
>
> They are technology-independent and apply to every Epic, document, engineering decision, and implementation.

---

# Rule 0 – Communication

- Team communication is conducted in **German**.
- All engineering artifacts are written in **English**.

Engineering artifacts include:

- Markdown documentation
- Source code
- Comments
- README files
- ADRs
- Standards
- Reports
- Cheat Sheets
- Templates
- Prompts
- GitHub Issues
- Pull Requests
- Commit messages
- Branch names

---

# Rule 1 – Repository First

The repository is the **Single Source of Truth**.

AI assistants generate content.

The repository defines the current state.

---

# Rule 2 – Explain → Decide → Continue

Every engineering discussion follows the same pattern.

1. Explain the problem.
2. Recommend one solution.
3. Make a decision.
4. Document the decision.
5. Continue.

Avoid unnecessary comparisons once a technically sound decision has been made.

---

# Rule 3 – One Decision → One Standard → One Cheat Sheet

Every relevant engineering decision produces exactly three artifacts.

- Decision (ADR)
- Standard
- Cheat Sheet (if required for daily work)

No duplicate documentation.

---

# Rule 4 – Every Discussion Produces an Artifact

Knowledge must not remain inside conversations.

Every completed discussion should produce one or more repository artifacts.

Examples:

- Standard
- ADR
- Report
- Handbook chapter
- Cheat Sheet
- Template
- Prompt

---

# Rule 5 – Documentation is Part of Engineering

Documentation is created **during engineering**, not afterwards.

Documentation is considered part of the deliverable.

---

# Rule 6 – Prefer Simplicity

Choose the simplest solution that satisfies the project's requirements.

Avoid unnecessary complexity.

Avoid premature optimization.

---

# Rule 7 – 80/20 Rule

Document first the knowledge that is used most frequently.

Rare edge cases can be added later.

---

# Rule 8 – NOW / NEXT / LATER

Every recommendation should belong to one category.

## NOW

Implement immediately.

## NEXT

Implement during the next logical step.

## LATER

Intentionally postpone until it creates value.

---

# Rule 9 – Pragmatism over Perfection

The goal is a working engineering platform.

Not a perfect enterprise solution.

---

# Rule 10 – Consistency over Completeness

A smaller, consistent Knowledge Base is more valuable than a larger, inconsistent one.

---

# Rule 11 – Reuse Before Creating

Before creating a new document, verify whether the information belongs in an existing one.

Avoid redundancy.

---

# Rule 12 – Living Knowledge Base

The Engineering Knowledge Base evolves together with the project.

It is continuously reviewed, improved, and maintained.

It is never considered "finished".

---

# Engineering Motto

> **Build once. Learn forever.**

---

# Documentation Motto

> **Document once. Reuse everywhere.**

---

# Hepberger's Rule

> **If experience solved the problem, capture it before it becomes memory.**
